# Pre-Evidence Integration Note

> **Provenance:** `collaborative` — Human-directed AI drafting with steward synthesis, revision, and final approval. See [Content Provenance Standard](CONTENT_PROVENANCE.md).

## Purpose

This note states, **before any meaningful evidence has arrived**, how the project will interpret what its different work tracks produce. It is a precommitment, written for one specific reason: to remove the project's own future freedom to read ambiguous or weak results as if they confirmed the theory.

The risk it guards against is concrete and was named during [Exchange #7 (Proof-of-Usefulness — Feedback Timescale Review)](../agent/exchanges/proof-of-usefulness-feedback-timescale-review.md): once a project is invested in a hypothesis, ambiguous results tend to get retroactively cited as "partial validation." The defense is to fix the interpretation rules in advance, in public, so that a later favorable reading has to argue against a standard the project set when it had nothing to gain from gaming it.

This is the [Roadmap Recommendation 5](../ROADMAP_ARCHIVE.md#pre-evidence-integration-note-recommendation-5) commitment (now complete and archived), and it is part of the same trust posture as the [Content Provenance Standard](CONTENT_PROVENANCE.md). It expresses [Principle 14 (truth and evidence as public goods)](../PRINCIPLES.md#14-truth-and-evidence-must-be-protected-as-public-goods) and [Principle 4 (accountable, legible power)](../PRINCIPLES.md#4-power-must-remain-accountable-legible-and-reversible) applied to the project's own claims.

## What each track produces — and what it cannot

The project runs distinct kinds of work that produce distinct kinds of evidence. Conflating them is the core error this note prevents. Each track is scoped to what it can actually support.

| Track | What it produces | What it does **not** establish |
|---|---|---|
| **Practitioner critique** (e.g. [Exchange #17](../agent/exchanges/first-practitioner-critique-ai-provenance-exchange.md), [#22](../agent/exchanges/entry-trust-domain-expert-practitioner-exchange.md)) | Evidence about whether the analysis is *useful, accurate, and recognizable* to people with domain experience | Any causal claim that a reform works. Critique tests the **analysis**, not the intervention. |
| **Fast-feedback case** (a bounded reform pilot) | Directional evidence on **execution** (can the thing be built and run) and on **behavioral trust proxies** (do people use it, return, refer others) | **Proof of recursive uplift.** A working pilot shows the execution layer is real; it does not show that competence cascades into trust, or trust into sequence. |
| **Cross-lineage / adversarial review** ([protocol](../agent/process/adversarial-review-protocol.md), [harness](../agent/process/cross-lineage-review-harness-protocol.md)) | Evidence about whether a claim **survives challenge** from independent reasoning | External-world validity. Surviving review is necessary, not sufficient; convergence among models is not evidence about reality. |
| **Historical parallel test** ([protocol](../agent/process/historical-parallel-test-protocol.md)) | Evidence about whether a reform sequence has **precedent** and how structurally similar cases fared | Predictive certainty for the present case. Reference-class fit is itself a judgment call. |

## Interpretation commitments

The project commits, in advance, to the following:

1. **Practitioner critique tests analytical usefulness, not causal claims.** A practitioner telling us the housing analysis is wrong, incomplete, or unrecognizable is strong evidence about the *analysis*. A practitioner finding it useful is **not** evidence that any proposed reform would succeed.

2. **A fast-feedback case produces directional evidence, not proof of the theory.** It speaks to the **execution layer** and to **behavioral trust proxies** only. It will not be cited as evidence for recursive uplift as a whole.

3. **Ambiguous results will be reported as ambiguous.** Where a result admits more than one reading, the project will present the readings side by side — including the reading least favorable to the project's hypothesis — rather than selecting the favorable one. Ambiguity is a finding, not a problem to be narrated away.

4. **Weak or null results will not be upgraded after the fact.** A result that was not designed to test a claim will not later be presented as if it had tested it. Pre-registered expectations (what would count as success, ambiguity, or failure) are set before the result is in, wherever feasible.

5. **The project will use its standing confidence vocabulary** — *established by evidence · working hypothesis · speculative · contested*, defined in the [Adversarial Review Protocol](../agent/process/adversarial-review-protocol.md) — to label every claim that evidence touches, and will downgrade as readily as it upgrades.

## The recursive-uplift layer caveat

Recursive uplift, the project's central mechanism hypothesis, decomposes into three layers that operate on **different timescales and under different evidentiary conditions** (the core finding of [Exchange #7](../agent/exchanges/proof-of-usefulness-feedback-timescale-review.md)). The project commits to keeping evidence matched to the layer it can actually reach:

- **Layer 1 — execution** (can a bounded improvement be built and measured): **testable on short timelines.** Fast-feedback cases speak here.
- **Layer 2 — trust** (does visible competence raise warranted trust): **partially observable through behavioral proxies**, but not reliably measurable through short-cycle surveys. Proxy evidence will be labeled as proxy evidence, with its attribution limits stated.
- **Layer 3 — sequence** (does trust enable the next reform, in order): **inherently long-cycle.** The project will **not** claim short-cycle evidence for the sequence layer. Sequencing effects unfold over years; any short-horizon result presented as evidence for Layer 3 would be a category error, and this note forecloses it in advance.

## Status

This note is the **interpretation half** of Recommendation 5. The **transparency half** — disclosing how content is authored and generated — is already live as the [Content Provenance Standard](CONTENT_PROVENANCE.md). With this note published, Recommendation 5 is complete.

It is a living commitment: as new tracks open (new pilot types, new review instruments), their evidence scope should be added to the table above *before* they produce results, not after.
